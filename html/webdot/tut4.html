In order to support client-side maps, WebDot can filter an html page and replace all occurrences of <webdot...> tags with a client-side map and an image reference. For example, tut4.dot contains: 
    graph G {
        graph [URL="/webdot/default.html"]
        node [URL="/webdot/\N.html"]
        a -- b -- c; b -- d;
    }
and the source of this page contains (for the graph on the right): 
    <webdot src=/webdot/tut4.dot engine=neato type=png target=new>

after serving this page through /cgi-bin/webdot, this <webdot...> tag is replaced by: 

    <map name="webdot0">
    <area href="/cgi-bin/webdot/http://127.0.0.1/webdot/tut4.dot.neato.png.help" shape="rect" coords="160,223,248,235" target="new">
    <area href="/webdot/a.html" shape="rect" coords="1,126,73,174" target="new">
    <area href="/webdot/b.html" shape="rect" coords="101,101,173,149" target="new">
    <area href="/webdot/c.html" shape="rect" coords="173,173,245,221" target="new">
    <area href="/webdot/d.html" shape="rect" coords="128,2,200,50" target="new">
    <area href="/webdot/default.html" shape="default" target="new">
    </map>
    <img src="/cgi-bin/webdot/http://127.0.0.1/webdot/tut4.dot.neato.png" width="248" height="235" usemap="#webdot0" border="0" alt="Graph by WebDot">

The advantages of this mode of WebDot are the much simpler authoring and the responsiveness of client-side mapping. Notice that the URL that would be reached by clicking on a node is flashed into the status bar at the bottom of the browser.

The disadvantage is that the page has to be served through the /cgi-bin/webdot program, and that care must be taken that relative URLs in the html don't also get served through the program. It is important to use "rooted" hrefs in documents served this way.
